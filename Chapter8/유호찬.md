# 장치 컨트롤러와 장치 드라이버

1. 입출력장치에는 종류가 굉장히 많다.
    
    키보드, 모니터, usb, ssd, 마우스, 프린터, 스피커, 마이크 등등
    
    다양한 입출력장치와 정보를 주고받는 방식을 규격화하기 어려움
    
2. cpu와 메모리는 데이터 전송률이 높지만 입출력 장치의 데이터 전송률은 낮다
    
    데이터 전송률 차이 때문에 cpu와 메모리, 입출력장치 간의 통신이 어려움 
    

위와 같은 이유로 인하여 입출력장치는 컴퓨터에 직접 연결되지 안혹 장치 컨트롤러를 통해 연결되고 장치 드라이버가 필요하다.

### 장치 컨트롤러

CPU와 입출력장치를 연결하기 위한 하드웨어적인 통로 

- CPU와 입출력장치 간의 통신 중개
- 오류 검출
- 데이터 버퍼링

장치 컨트롤러가 일종의 번역가 역할을 하고 오류를 검출한다.

데이터 버퍼링 : 전송률이 다른 장치 사이에 버퍼라는 저장 공간에 데이터를 저장하여 전송률을 비슷하게 맞추는 방법이다.

<img width="935" height="744" alt="스크린샷 2026-04-09 191719" src="https://github.com/user-attachments/assets/fbec855e-b5d8-4719-9579-5a2a08b621be" />

장치 컨트롤러 내부에는 데이터, 상태, 제어 레지스터가 존재한다.

이 레지스터에 담긴 값들은 버스를 타고 CPU나 다른 입출력장치로 전달되기도 하고, 연결된 입출력장치로 전달된다. 

1. 데이터 레지스터
    
    CPU와 입출력장치 사이에 주고받을 데이터가 담기는 레지스터
    
2. 상태 레지스터
    
    입출력 장치의 상태 정보를 저장 (입출력 작업 준비, 완료, 오류 등)
    
3. 제어 레지스터
    
    입출력장치가 수행할 내용에 대한 제어 정보와 명령을 저장 
    

### 장치 드라이버

장치 컨트롤러의 동작을 감지하고 제어함으로써 장치 컨트롤러가 컴퓨터 내부와 정보를 주고받을 수 있게 하는 프로그램

소프트웨어적인 통로이다.

# 다양한 입출력 방법

입출력 작업을 수행하려면 CPU와 장치 컨트롤러가 정보를 주고받아야한다.

크게 세가지 방법이 있다. 

1. 프로그램 입출력
2. 인터럽트 기반 입출력
3. DMA 입출력

### 프로그램 입출력

프로그램 속 명령어로 입출력 장치를 제어하는 방식

프로그램의 명령어를 실행하는 과정에서 입출력 명령어를 만나면 입출력장치와 연결된 장치 컨트롤러와 상호작용하며 입출력 작업을 수행

CPU 내부 레지스터와 달리 장치 컨트롤러 속 레지스터 주소를 알기 위해 메모리 맵 입출력과 고립형 입출력 두가지 방법이 사용된다.

1. 메모리 맵 입출력
    
    메모리에 접근하기 위한 주소 공간과 입출력장치에 접근하기 위한 주소 공간을 하나의 주소 공간으로 간주하는 방법
    
    장치 컨트롤러의 레지스터도 똑같이 메모리 주소 대하듯 하면 된다.

### 인터럽트 기반 입출력

장치 컨트롤러에게 입출력장치 작업을 지시한 뒤 CPU는 다른 작업을 할 수 있다. 

이후 장치 컨트롤러가 입출력장치 작업이 끝났다면 CPU에게 인터럽트 요청 신호를 보내서 CPU는 하던 작업을 멈추고 인터럽트 서비스 루틴을 실행해서 입출력장치 작업을 한다.

<img width="1062" height="465" alt="스크린샷 2026-04-13 102618" src="https://github.com/user-attachments/assets/fd054b9e-c6d0-4330-8914-ea96714cac38" />

만약 여러가지 인터럽트가 동시다발적으로 일어난다면 우선순위를 고려해서 우선순위가 높은 인터럽트부터 처리합니다.

플래그 레지스터 속 인터럽트 비트가 활성화되어 있거나 인터럽트 비트를 비활성화해도 무시할 수 없는 인터럽트인 NMI가 발생한 경우 우선순위가 높은 인터럽트부터 처리한다.

<img width="1042" height="591" alt="스크린샷 2026-04-13 113705" src="https://github.com/user-attachments/assets/426c2b22-ffe2-42cc-82b1-a8416ac22332" />

많은 컴퓨터에서는 프로그래머블 인터럽트 컨트롤러(PIC)라는 하드웨어를 사용하여 인터럽트의 우선순위를 판별한다.

여러가지 인터럽트를 받아서 우선순위를 판단하고 CPU에게 가장 먼저 처리할 인터럽트를 알려준다.

<img width="917" height="375" alt="스크린샷 2026-04-13 113957" src="https://github.com/user-attachments/assets/01575c2a-9a41-47d4-81bf-946c61e5a261" />

PIC의 처리 과정

1. PIC가 장치 컨트롤러에서 인터럽트 요청 신호들을 받아들인다
2. 인터럽트 우선순위를 판단한 뒤 CPU에게 인터럽트 요청 신호를 보낸다
3. CPU는 PIC에 인터럽트 확인 신호를 보낸다.
4. PIC는 데이터 버스를 통해 CPU에 인터럽트 벡터를 보낸다
5. CPU는 인터럽트 벡터를 통해 요청의 주체를 알고 해당 장치의 인터럽트 서비스 루틴을 실행한다

### DMA 입출력

DMA(Direct Memory Access)는 이름 그대로 메모리에 직접 접근할 수 있는 입출력 기능이다.

DMA 입출력을 하기 위해서는 시스템 버스에 연결된 DMA 컨트롤러라는 하드웨어가 필요하다.

<img width="1299" height="536" alt="스크린샷 2026-04-13 120112" src="https://github.com/user-attachments/assets/464d1d6b-4a79-4403-b5b0-e84b3ee78a29" />

DMA 입출력 과정

1. CPU는 DMA 컨트롤러에 입출력장치의 주소, 수행할 연산, 읽거나 쓸 메모리 주소 등과 같은 정보로 입출력 작업을 명령
2. DMA 컨트롤러는 CPU 대신 장치 컨트롤러와 상호작용하며 입출력 작업을 수행. 이떄 필요한 경우 DMA 컨트롤러가 직접 메모리에 접근
3. 입출력 작업이 끝나면 DMA 컨트롤러는 CPU에 인터럽트를 건다.

이러한 과정을 통해 CPU는 DMA 컨트롤러에 입출력 작업 명령만 내리고 입터럽트만 받으면 된다. 입출력의 시작과 끝에만 관여한다.

시스템 버스는 동시 사용이 불가하기 때문에 CPU가 사용하지 않을 때마다 조금씩 DMA 컨트롤러가 시스템버스를 사용하여 메모리에 접근한다.

### 입출력 버스

DMA 컨트롤러가 시스템 버스를 많이 사용할 수록 CPU가 시스템 버스를 이용하지 못한다. 이를 해결하기 위해 DMA 컨트롤러와 장치 컨트롤러들을 입출력 버스라는 별도의 버스에 연결했다.

장치 컨트롤러들이 시스템 버스가 아닌 입출력 버스로 DMA 컨트롤러와 연결하여 시스템 버스 사용 빈도를 줄인다.
