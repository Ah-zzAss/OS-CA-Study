# Chapter8: 입출력장치

## 1. 입출력 장치 연결

입출력 장치를 연결하기 위해 **하드웨어적인 통로인 장치 컨트롤러**와 **소프트웨어적인 통로인 장치 드라이버**가 존재한다.

<img width="366" height="280" alt="image" src="https://github.com/user-attachments/assets/16279fee-8f8f-4c69-8581-07bca76528c1" />

### 1. 장치 컨트롤러

**컴퓨터 내부와 정보를 주고 받기 위해 사용하는 하드웨어**  
_입출력 제어기, 입출력 모듈이라고 불리기도 한다._

#### (1) 등장 배경

1. 장치의 종류는 다양하고,  장치마다 속도, 데이터 전송 형식 등도 다양하다.
2. CPU와 메모리의 데이터 전송률은 높지만 입출력 장치의 데이터 전송률은 낮다.

> * 전송률: 데이터를 얼마나 빠르게 교환할 수 있는지 나타내는 지표

#### (2) 기능

1. **CPU와 입출력장치 간의 통신 중개**
2. **오류 검출**
3. **데이터 버퍼링**
    - 전송률이 높은 장치와 낮은 장치 사이에 주고 받는 데이터를 **버퍼**(임시 저장 공간)에 저장하여 **전송률을 비슷하게 맞추는 방법**
    - 모았다가 한 번에 내보내거나 or 모았다가 조금씩 내보내는 방법

#### (3) 구조

- **데이터 레지스터**: CPU와 입출력장치 사이에 주고 받을 데이터를 저장  
    _(최근 주고 받는 데이터가 많은 입출력 장치에서는 레지스터 대신 RAM을 사용하기도 한다.)_
- **상태 레지스터**: 입출력장치의 입출력 작업 준비 여부, 작업 완료 여부, 오류 검출 등의 상태 정보 저장
- **제어 레지스터**: 입출력장치가 수행할 내용에 대한 제어 정보와 명령

### 2. 장치 드라이버

장치 컨트롤러의 동작을 감지, 제어함으로써 **장치 컨트롤러가 컴퓨터 내부와 정보를 주고 받을 수 있게** 하는 프로그램

## 2. 다양한 입출력 방법

| 구분 | 핵심 포인트 (동작 방식) | CPU의 관점 |
| :--- | :--- | :--- |
| **프로그램 입출력** | CPU가 장치 컨트롤러의 레지스터를 주기적으로 확인(Polling)하며 데이터를 직접 주고받음 | 입출력이 완료될 때까지 다른 작업을 하지 못하고 대기함 |
| **인터럽트 입출력** | 입출력 장치가 준비되면 **인터럽트 신호**를 보내 CPU에 알림. CPU는 요청 시에만 개입함 | 입출력 요청 후 다른 작업을 수행하다가 신호가 올 때만 처리함 |
| **DMA 입출력** | **DMA 컨트롤러**가 CPU를 거치지 않고 메모리와 장치 간에 데이터를 직접 전송함 | 전송 시작과 종료 시에만 관여하며 시스템 버스 사용권을 잠시 양보함 |

### 1. CPU의 주소 할당 방식

CPU는 **CPU 주소 체계**를 통해 **장치 컨트롤러의 레지스터 주소 공간으로 접근** 가능하다.

| 구분 | 메모리 맵 입출력 (Memory-mapped I/O) | 고립형 입출력 (Isolated I/O) |
| :--- | :--- | :--- |
| **주소 공간** | 메모리와 입출력 장치가 **하나의 주소 공간**을 공유함 | 메모리와 입출력 장치의 **주소 공간이 분리**되어 있음 |
| **명령어** | 메모리 접근 명령어(`MOV`, `LOAD` 등)를 **공통 사용** | 입출력 전용 명령어(`IN`, `OUT` 등)를 **따로 사용** |
| **핵심 특징** | **메모리 주소 공간의 일부가 축소됨** | **별도의 입출력 제어 신호와 명령어가 필요함** |

#### (1) 메모리 맵 입출력 (memory-mapped I/O)

<img width="277" height="232" alt="image" src="https://github.com/user-attachments/assets/5c36ad2d-d0ce-4b6f-b1d2-7957684778b8" />

- 다음의 두 주소 공간을 **하나의 공간**으로 간주하는 방법 (메모리 주소 공간 축소)
    1. **메모리**에 접근하기 위한 주소 공간
    2. **입출력 장치**에 접근하기 위한 주소 공간

- CPU는 메모리 주소들이나 장치 컨트롤러의 레지스터들이나 **모두 똑같이 메모리 주소를 대하듯** 동작한다.
- CPU가 '517번지를 읽어라'라는 명령어를 실행했을 때,
    - 517번지가 메모리 상의 주소를 가리킨다면 메모리 517번지에 저장된 정보를 읽어들인다.
    - 517번지가 프린터 컨트롤러의 상태 레지스터를 가리킨다면 프린터의 상태를 확인한다.

#### (2) 고립형 입출력 (Isolated I/O)

<img width="613" height="220" alt="image" src="https://github.com/user-attachments/assets/9803dc10-9fbc-4b43-b5f7-3396c5519345" />

- 다음의 두 주소 공간을 **분리하는** 방법 (메모리 주소 공간 유지)
    1. 메모리에 접근하기 위한 주소 공간
    2. 입출력 장치에 접근하기 위한 주소 공간

- CPU는 입출력장치에 접근하기 위해 **메모리에 접근하는 명령어와는 다른** 입출력 명령어를 사용합니다.

### 2. 인터럽트 기반 입출력

장치 컨트롤러가 입출력 작업을 끝낸 뒤 CPU에게 인터럽트 요청 신호를 보내면  
CPU는 **하던 일을 잠시 스택에 백업**하고 **인터럽트 서비스 루틴을 실행**한다.

<img width="658" height="241" alt="image" src="https://github.com/user-attachments/assets/99f25cc9-3f9b-47de-a112-39b351dcfa33" />

> * (비교되는 개념) '폴링': 주기적으로 입출력장치의 상태를 확인하는 방식

#### (1) 다중 인터럽트의 처리

CPU는 인터럽트 간에 우선순위를 고려하여 우선순위가 높은 순대로 처리할 수 있다.

<img width="677" height="225" alt="image" src="https://github.com/user-attachments/assets/65c2951c-fee7-4166-9247-efb63c9fc8a9" />

- 플래그 레지스터 속 인터럽트 비트가 **활성화** 되어있는 경우 (인터럽트 수용 가능 상황) 
- 플래그 레지스터 속 인터럽트 비트가 비활성화 되어있지만, 무시할 수 없는 **NMI(Non-Maskable Interrupt)**가 발생한 경우

#### (2) 다중 인터럽트를 처리하는 방법

**프로그래머블 인터럽트 컨트롤러 (PIC, Programmable Interrupt Controller)** 라는 **하드웨어**를 사용한다.

<img width="1418" height="708" alt="image" src="https://github.com/user-attachments/assets/17408331-0317-4c38-8530-46daaa118e1d" />

- 여러 장치 컨트롤러에 연결되어 있다.
- 여러 하드웨어 인터럽트 요청들의 **우선순위 판별**하여 CPU에게 처리할 인터럽트를 알려준다.
- **NMI는 우선 순위가 가장 높아 판별이 불필요**하므로 PIC가 판별하지 않음

### 3. DMA 입출력

Direct Memory Access, 
**입출력장치와 메모리가 CPU를 거치지 않고 상호작용**할 수 있는 방식

- **등장 배경**: 입출력 장치와 메모라 사이에 전송되는 모든 데이터가 CPU를 거쳐야한다.
<img width="694" height="208" alt="image" src="https://github.com/user-attachments/assets/ebe6c690-4988-4443-accb-71768cea2bb3" />

- **방법**: 시스템 버스에 연결된 **DMA 컨트롤러**라는 하드웨어를 통하여 동작한다.

#### (1) DMA 입출력 과정

**CPU는 DMA 컨트롤러에 명령만 내리고, 인터럽트만 받는다.**
입출력장치와 메모리 사이에 주고 받을 **데이터는 CPU를 거치지 않는다.**

<img width="642" height="251" alt="image" src="https://github.com/user-attachments/assets/39aa5b1f-db33-4523-b2c9-46b31dbd835e" />

> **[세부 과정]**  
    1. CPU가 DMA 컨트롤러에 백업할 내용이 저장된 메모리 주소 등의 입출력 작업을 명령한다.  
    2. DMA 컨트롤러는 메모리와 직접 상호작용하며 백업할 정보를 가져온다.  
    3. DMA 컨트롤러는 하드디스크의 장치 컨트롤러에 내보낸다.  
    4. 백업이 끝나면 DMA 컨트롤러는 CPU에게 인터럽트를 걸어 작업이 끝났음을 알린다.

#### (2) 입출력 버스

DMA 컨트롤러와 장치 컨트롤러가 서로 데이터 전송을 위해 사용하는 버스,
이를 통해 시스템 버스의 이용 빈도를 낮출 수 있다.

<img width="612" height="246" alt="image" src="https://github.com/user-attachments/assets/adb7bfbc-4d83-4aa5-b91f-e59a7231f4cc" />

- **등장 배경**: **사이클 스틸링(Cycle Stealing)**, DMA를 위해 시스템 버스를 자주 사용하면 그만큼 CPU가 시스템 버스를 이용하지 못하는 상황이 발생  
    (시스템 버스는 공용 자원이기 때문에 동시 사용이 불가능)
