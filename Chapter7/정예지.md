# Chapter7: 보조기억장치

| 구분 | 하드 디스크 (HDD) | 플래시 메모리 (SSD/USB) |
| :--- | :--- | :--- |
| **저장 방식** | **자기적** 방식 (플래터 회전) | **전기적** 방식 (반도체 소자) |
| **수정 방식** | **덮어쓰기(Overwrite) 가능** | **덮어쓰기 불가능** (삭제 후 쓰기) |
| **속도** | 상대적으로 느림 (기계적 이동 시간) | 매우 빠름 (전기적 신호) |
| **내구성** | 충격에 약함 (소음/발열 존재) | 충격에 강함 (무소음/저발열) |

## 1. 하드 디스크

**자기적인 방식으로 데이터를 저장하는 보조기억장치 (자기 디스크)**

<img width="350" height="auto" alt="image" src="https://github.com/user-attachments/assets/a0047add-ce9a-4734-bd32-fd3d2e1db344" />

### (1) 데이터에 접근하는 시간

- **탐색 시간 (seek time)**: 접근하려는 데이터가 저장된 트랙까지 헤드를 이동시키는 시간
- **회전 지연 (rotational latency)**: 헤드가 있는 곳으로 플래터를 회전시키는 시간 
- **전송 시간 (transfer time)**: 하드 디스크와 컴퓨터 간에 데이터를 전송하는 시간

### (2) 데이터 접근 시간을 단축하는 방법

- 탐색 시간, 회전시간을 단축 시키기 위해 플래터를 빠르게 돌려 RPM을 높이는 것
- 참조 지역성, 접근하려는 데이터가 플래터 or 헤드를 조금만 옮겨도 접근할 수 있는 곳에 위치해있는 것

### (3) 종류

- **단일 헤드 디스크 (이동 헤드 디스크)**: 플래터의 한 면당 헤드가 하나씩 달려있는 하드 디스크
- **다중 헤드 디스크 (고정 헤드 디스크)**: 헤드가 트랙별로 달려있는 하드 디스크

- 다중 헤드 디스크는 탐색 시간이 0이다.

## 2. 플레시 메모리

**전기적으로 데이터를 읽고 쓸 수 있는 반도체 기반의 저장 장치**
(범용성이 넓어 보조기억장치에만 속한다고 보기는 어렵다.)
_예시: USB 메모리, SD 카드, SSD, ROM에도 사용된다._

<img width="350" height="auto" alt="image" src="https://github.com/user-attachments/assets/5320ed9b-70a0-44a1-8bb4-f3376a24cc0b" />

### (1) 종류

**하나의 셀에 몇 비트를 저장할 수 있냐에 따라 종류가 나뉜다.**
(셀 : 플래시 메모리에서 데이터를 저장하는 가장 작은 단위)
```
SLC(Single Level Cell)
-> MLC (Mutiple Level Cell)
-> TLC (Triple Level Cell)
```
- 한 셀에 n비트를 저장할 수 있다면 2^n가지의 정보를 표현할 수 있다.
- 한 셀에 저장할 수 있는 비트 수가 많아질 수록
    - 수명이 짧아진다.
    - 용량 대비 가격이 저렴해진다.

### (2) 읽기/쓰기/삭제 단위

플래시 메모리는 **'데이터 삭제'가 '데이터 쓰기'보다 큰 단위로만 가능하다.**
- **읽기/쓰기**: 페이지 단위
- **삭제**: 블록 단위

> <img width="731" height="254" alt="image" src="https://github.com/user-attachments/assets/8b60e69c-1ff6-4562-bf9e-547ac0b4aff4" />
> - **페이지(page)**: cell들이 모여 만들어진 단위  
> - **블록(block)**: page가 모여 만들어진 단위  
> - 플레인(Plane): 여러 블록이 모인 단위  
> - 다이(Die): 여러 플레인이 모인 하나의 독립적인 반도체 칩  

#### 페이지가 가질 수 있는 상태

- **Free**: 새로운 데이터를 저장할 수 있는 상태 (어떠한 데이터를 저장하고 있지 않음)
- **Valid**: 이미 유효한 데이터를 저장하고 있는 상태
- **Invalid**: 유효하지 않은 데이터(쓰레기값)을 저장하고 있는 상태

#### 가비지 컬렉션 (Garbage Collection)

**유효한 페이지만을 새로운 블록으로 복사**한 뒤, **기존의 블록을 삭제**하는 기능  

<img width="414" height="151" alt="image" src="https://github.com/user-attachments/assets/94f785fe-96ba-4331-a2ca-b40ba429af34" />

**[수행 과정]**
1. 기존 블록에 Invalid 페이지가 많아져서 저장 공간이 부족
2. 해당 블록 내의 Valid 페이지만 골라내어 다른 빈 블록(Free 상태)으로 복사
3. 유효한 데이터를 옮긴 후, 기존의 블록을 통째로 삭제한다.
4. 삭제된 블록은 다시 모든 페이지가 Free 상태가 되어 새로운 데이터를 쓸 수 있게 된다.


## 3. RAID
(Reduntdant Array of Independent Disks)  

**여러 개의 물리적 보조기억장치를 마치 하나의 논리적 보조기억 장치처럼 사용하는 기술**

- **목적**: 데이터의 안정성, 높은 성능 (보조기억장치는 수명이 있다.)
- **사용**: 하드 디스크, SSD

> <img width="673" height="219" alt="image" src="https://github.com/user-attachments/assets/45654f08-98b9-4e58-8c08-695691f572a6" />
> 1TB 하드디트크 4개로 RAID를 구성하면 4TB 하드 디스크 한 개의 성능과 안정성을 능가할 수 있다.

### (1) RAID 레벨 (RAID 구성 방법)

#### RAID 0

**여러 개의 보조기억장치에 데이터를 단순히 나누어 저장**하는 구성 방식  

<img width="683" height="246" alt="image" src="https://github.com/user-attachments/assets/f77cf0fb-d161-4f02-8279-4c33081e261a" />

**장점**: 저장된 데이터를 읽고 쓰는 것이 빨라진다.  
  - 하나의 저장 장치: a1~a4 데이터를 여러 번에 걸쳐 읽는다.  
  - 여러 개의 저장 장치: a1~a4 데이터를 각 데이터베이스에 한 개씩 저장한다.  

**단점**: 구성되는 하드 디스크 중 하나에 문제가 생긴다면 다른 모든 하드 디스크의 정보를 읽는데 문제가 생긴다.  

> - stripe: 줄무늬처럼 분산된 데이터
> - striping: 분산하여 저장하는 것

#### RAID 1

**복사본**을 만든다. (mirroring)
<img width="697" height="267" alt="image" src="https://github.com/user-attachments/assets/a6e988e7-7c62-4f43-a348-517eaceff297" />

**장점**: 복구가 간단하다.  
**단점**: 
- 데이터를 쓸 때는 두군데에 쓰기 때문에 RAID 0보다 느리다
- 복사본이 만들어지는 만큼 사용 가능한 용량이 적어진다.

#### RAID 4

**오류를 검출하고 복구하기 위한 정보(parity bit)를 저장**한 장치를 두는 구성 방식  

<img width="697" height="235" alt="image" src="https://github.com/user-attachments/assets/841f9f38-d305-438b-9029-c918dd02cfa3" />

- parity를 저장한 장치를 이용하여 다른 장치들의 오류를 검출하고, 오류가 있다면 복구한다.

    > 원래 parity bit는 오류 검출용 정보이다.  
    그러나 RAID에서는 패리티 값으로 오류 수정도 가능하다.

- **장점**: RAID 1보다 적은 하드 디스크로도 데이터를 안전하게 보관 가능
- **단점**: 패리티 디스크의 병목
    - RAID 4에서 새로운 데이터가 저장될 때마다 parity를 저장하는 디스크에도 쓰게 되므로 parity 저장 장치에 병목 발생
    <img width="669" height="250" alt="image" src="https://github.com/user-attachments/assets/1fe1ac58-974e-4ade-a38d-603e5bc7cd90" />

#### RAID 5

**패리티 정보를 분산하여 저장**하는 방식
  
<img width="635" height="245" alt="image" src="https://github.com/user-attachments/assets/21ce3541-e2ab-422a-90f9-813c90a7ad2d" />

#### RAID 6

기본적으로 RAID 5와 같으나, **서로 다른 두 개의 parity를 두는 방식**

<img width="662" height="246" alt="image" src="https://github.com/user-attachments/assets/cac5d4a5-4fa9-4247-960c-037ff0f1e186" />

- 데이터 저장 속도를 조금 희생하더라도 안전하게 보관하고 싶을 때 사용
- 오류를 검출, 복구할 수 있는 수단이 2개
- 함께 저장할 parity가 2개이므로 쓰기 속도는 RAID 5보다 느리다.

#### Nested RAID

여러 RAID레벨을 혼합한 방식

- RAID 10: RAID 0 + RAID 1
- RAID 50: RAID 0 + RAID 5
